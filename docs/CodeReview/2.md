## Findings

### 【严重-1】`auth/auth_login.rs:93` — PKCE `code_verifier` 明文写入事件日志

**文件**: `crates/service/src/auth/auth_login.rs` 第 93 行

**问题原因**: `login_start()` 函数在写入事件日志时将 PKCE `code_verifier` 以明文形式序列化到 `message` 字段：
```rust
message: format!(
    "{{\"login_id\":\"{}\",\"code_verifier\":\"{}\"}}",
    state, pkce.code_verifier
),
```
`code_verifier` 是 OAuth PKCE 流程的核心机密，等同于对 `code_challenge` 的解锁钥匙。

**潜在影响**: 任何能读取数据库 `events` 表或通过 RPC 查看事件日志的人，都能拿到 `code_verifier`，配合拦截到的 `authorization_code` 可完成 token exchange，接管账号。

**建议修复**: 从事件 message 中移除 `code_verifier`，仅保留 `login_id`。

### 【严重-2】`account/account_export.rs:210-218` — 导出账号时敏感令牌以明文 JSON 输出

**文件**: `crates/service/src/account/account_export.rs` 第 210–218 行

**问题原因**: `build_account_export_json()` 直接将 `access_token`、`id_token`、`refresh_token` 写入导出 JSON。虽然数据库内部已通过 AES-256-GCM 加密存储，但导出操作先解密再明文序列化，完全绕过了加密保护。

**潜在影响**: 导出文件可被任何拥有文件系统读权限的人用于接管所有已登录账号。

**建议修复**: 导出时使用独立的加密封装（如用户提供的导出密码 + PBKDF2/Argon2 派生密钥），或至少在文档和 UI 中明确告警导出包含敏感凭据。

### 【严重-3】`crates/core/src/crypto.rs:58-62` — 加密失败时静默返回明文

**文件**: `crates/core/src/crypto.rs` 第 58–62 行

**问题原因**: `encrypt_field()` 在 AES-256-GCM 加密操作失败时（`Err(_)`）直接返回原始明文：
```rust
Err(_) => plaintext.to_string(),
```

**潜在影响**: 若加密意外失败（OOM、密钥损坏等），令牌将以明文形式写入数据库，且调用方无法感知。后续解密路径会把无前缀数据视为"旧格式明文"透传，掩盖了加密失效的事实。

**建议修复**: 将 `encrypt_field` 返回 `Result<String, ...>`，让调用方决定是否允许降级。至少应在失败分支记录 `log::error!`。

---

### 【高危-1】`auth/rpc.rs:8-9` + `runtime/process_env.rs:89` + `app_settings/env_overrides/process.rs:47-54` — 多线程环境下调用 `std::env::set_var`

**文件**:
- `crates/service/src/auth/rpc.rs` 第 8、9 行
- `crates/service/src/runtime/process_env.rs` 第 89 行（`load_env_from_exe_dir`）
- `crates/service/src/app_settings/env_overrides/process.rs` 第 47–54 行

**问题原因**: `std::env::set_var()` 和 `std::env::remove_var()` 在 Rust 1.66+ 中被标记为 `unsafe`（即将在未来版本成为编译错误）。这些调用发生在多线程上下文中（HTTP worker pool、tokio runtime 已启动），构成未定义行为。

**潜在影响**: 内存安全性破坏。可能导致进程崩溃、数据损坏或安全缺陷。env_overrides 模块尤为严重，因为它在运行时被用户主动触发。

**建议修复**: 使用进程内的 `HashMap<String, String>` 或专用配置结构体替代 `std::env::set_var`。仅在进程启动的单线程阶段设置环境变量，或使用 `unsafe` 块并加上详细的 safety 注释。

### 【高危-2】`crates/core/src/crypto.rs:119-128` — 密钥文件在 Windows 上无 ACL 保护

**文件**: `crates/core/src/crypto.rs` 第 119–128 行

**问题原因**: `load_or_create_key()` 仅在 `#[cfg(unix)]` 下设置 `0o600` 权限，Windows 平台下密钥文件以默认 ACL 创建，通常继承父目录权限，同机其他用户可能具有读取权限。

**潜在影响**: 在多用户 Windows 环境中，其他用户可读取密钥文件内容，进而解密数据库中所有加密字段。

**建议修复**: 添加 `#[cfg(windows)]` 分支，使用 Windows ACL API 限制密钥文件仅对当前用户可读写。

---

### 【高危-3】`auth/web_access.rs:51` — Web 会话令牌使用 SHA-256 而非 HMAC

**文件**: `crates/service/src/auth/web_access.rs` 第 51 行

**问题原因**: `build_web_access_session_token()` 使用 `SHA256("codexmanager-web-auth-session:{password_hash}:{rpc_token}")` 构造会话令牌。这是简单的拼接哈希，不是标准 HMAC 结构，易受长度扩展攻击影响（虽然 SHA-256 本身不易受影响，但这不是加密最佳实践）。更关键的是，`password_hash` 本身被直接用作 HMAC 的"密钥材料"——如果攻击者知道了 Argon2id 哈希字符串和 RPC token，便可伪造会话令牌。

**潜在影响**: 会话令牌可预测性高于使用 HMAC-SHA256 + 独立密钥的方案。

**建议修复**: 改用 `HMAC-SHA256(key=rpc_token, message=password_hash)` 或使用随机会话令牌并在服务端维护映射。

---

### 【高危-4】`runtime/process_env.rs:172-196` — RPC Token 文件在 Windows 上无 ACL 保护

**文件**: `crates/service/src/runtime/process_env.rs` 第 172–196 行

**问题原因**: `persist_rpc_token_if_missing()` 将 RPC 认证令牌写入文件，但没有设置文件权限。与 `crypto.rs` 中密钥文件的情况类似，Windows 上文件默认可被同机其他用户读取。

**潜在影响**: 拥有 RPC token 的攻击者可以完全控制 CodexManager 的后端 RPC 接口（读写账号、管理密钥、触发登录等）。

**建议修复**: 与密钥文件同样处理，在 Windows 上限制文件 ACL 为仅当前用户。

### 【中危-2】`gateway/mod.rs` — 手写 base64 解码器

**文件**: `crates/service/src/gateway/mod.rs`

**问题原因**: `decode_base64_standard()` 函数手工实现了一个 base64 standard 解码器（含 lookup table、padding 处理等）。项目已经依赖了 `base64` crate（在 `crypto.rs` 中使用）。

**潜在影响**: 手写解码器增加了维护和正确性风险。虽然当前实现看起来功能正确，但自行实现密码学/编码基础设施不是最佳实践。

**建议修复**: 替换为 `base64::engine::general_purpose::STANDARD.decode()`。

### 【中危-3】`app_settings/api/patch.rs` — 多处 `let _ =` 丢弃 Result

**文件**: `crates/service/src/app_settings/api/patch.rs`

**问题原因**: 设置更新函数中大量使用 `let _ =` 忽略子操作的 `Result`。包括对 gateway 运行时配置的修改（route strategy、proxy URL、compression 等），失败后无日志、无用户提示。

**潜在影响**: 用户通过 UI 修改设置后，部分子配置可能静默失败，造成运行时状态与用户预期不一致。

**建议修复**: 至少在每个 `let _ =` 处添加 `log::warn!` 记录失败。

### 【中危-4】`gateway/observability/trace_log.rs:44-47` — trace writer 线程名不含 `panic = "abort"` 保护

**文件**: `crates/service/src/gateway/observability/trace_log.rs` 第 44–47 行

**问题原因**: `TraceAsyncWriter::new()` 通过 `thread::Builder::new().spawn()` 创建一个后台写入线程，但写入循环内部若发生 panic，该线程将静默退出。`SyncSender` 后续写入会返回 `Err`（channel disconnected），但已入队的日志可能丢失。

**潜在影响**: trace 日志静默停止写入，排查问题时无法获取关键上下文。

**建议修复**: 在写入循环中包裹 `catch_unwind`，或注册 panic hook 进行告警。

### 【中危-5】`gateway/routing/selection.rs` — candidate 缓存使用 `Instant`（非持久化安全）

**文件**: `crates/service/src/gateway/routing/selection.rs`

**问题原因**: 候选人缓存使用 `std::time::Instant` 作为 TTL 基准。`Instant` 是单调时钟，不可序列化/持久化。虽然当前仅用于内存缓存，但如果未来尝试持久化缓存状态会导致问题。另外，在系统休眠（suspend/hibernate）后，`Instant` 在某些平台上的行为未明确定义。

**潜在影响**: 系统休眠/恢复后缓存 TTL 可能失真，导致使用过期的候选人列表。

**建议修复**: 当前为内存缓存场景可接受，但建议在休眠恢复时增加缓存失效逻辑。

---

### 【中危-6】`gateway/routing/request_gate.rs` — Condvar 锁中毒后返回错误而非恢复

**文件**: `crates/service/src/gateway/routing/request_gate.rs`

**问题原因**: 项目其他模块（如 `lock_utils.rs`）统一使用 `lock_recover()` 从 poison 中恢复，但 `request_gate.rs` 在 Condvar 等待时遇到 poison 后直接返回 `Err`，这意味着一个 panic 就能永久阻塞该 account 的后续请求。

**潜在影响**: 某个请求处理线程 panic 后，对应账号的所有后续请求都将因 gate 报错而失败。

**建议修复**: 与项目其他模块保持一致，使用 `into_inner()` 从 poison 中恢复。

---

### 【中危-7】`gateway/upstream/config.rs` — 环境变量直接控制上游代理 URL，无安全校验

**文件**: `crates/service/src/gateway/core/runtime_config.rs`

**问题原因**: `CODEXMANAGER_UPSTREAM_PROXY_URL` 和 `CODEXMANAGER_PROXY_LIST` 环境变量允许指定任意代理 URL。如果恶意用户能设置环境变量（或通过 env_overrides UI 配置），可将所有上游流量定向到攻击者控制的代理。

**潜在影响**: 中间人攻击，所有上游通信（含 OAuth 令牌、API 请求）都可被截获。

**建议修复**: 在 env_overrides 的 catalog 中，对 proxy URL 做 scheme 白名单校验（仅允许 `http://`、`https://`、`socks5://`）和潜在 SSRF 防护（禁止内网地址如 `169.254.x.x`、`10.x.x.x`）。

### 【中危-8】`app_settings/store.rs:7-12` — 每次读取配置都重新打开 Storage

**文件**: `crates/service/src/app_settings/store.rs` 第 7–12 行

**问题原因**: `open_app_settings_storage()` 每次调用都会创建新的 `Storage` 实例（即打开一次 SQLite 连接并初始化）。`sync_runtime_settings_from_storage()` 在启动时大量调用 `get_persisted_app_setting()`，每次调用都触发一次完整的打开 + init 流程。

**潜在影响**: 启动性能开销较大；在高并发 RPC 请求中如果有读取配置路径，会产生不必要的 SQLite 连接开销。

**建议修复**: 在 `sync_runtime_settings_from_storage()` 中一次性打开 Storage，传入各子函数复用；或使用 thread-local 缓存（类似 `storage_helpers.rs` 中的模式）。

### 【低危-1】`gateway/observability/request_log.rs:25-40` — 模型价格硬编码

**文件**: `crates/service/src/gateway/observability/request_log.rs` 第 25–40 行

**问题原因**: 模型单价以硬编码常量数组维护（`MODEL_PRICE_PER_1K_TOKENS`）。OpenAI 经常调价或新增模型，硬编码价格会很快过时。

**潜在影响**: 用量成本估算不准确。

**建议修复**: 将价格表外置为配置文件或远程更新源。当前作为估算可接受，注释已说明临时性质。

### 【低危-2】`runtime/lock_utils.rs` — 所有锁中毒一律恢复

**文件**: `crates/service/src/runtime/lock_utils.rs`

**问题原因**: `lock_recover()` / `read_recover()` / `write_recover()` 在遇到 `PoisonError` 时一律调用 `into_inner()` 恢复。这意味着即使某个线程在修改共享状态途中 panic 留下了不一致的数据，后续线程仍然会继续使用该数据。

**潜在影响**: 理论上可能在 panic 后使用不一致的共享状态。实际场景中 panic 通常在业务逻辑层被 `catch_unwind` 捕获，该风险较低。

**建议修复**: 添加注释说明设计决策。在关键数据结构（如金融相关计算）上考虑不恢复。

---

### 【低危-3】`gateway/request/request_rewrite.rs:100` — 手写 multipart boundary 解析

**文件**: `crates/service/src/gateway/request/request_rewrite.rs` 第 100 行附近

**问题原因**: `find_subsequence` 和 `extract_multipart_part_name` 手工解析 multipart 消息的 boundary 和 part name，未使用成熟的 multipart 解析库。

**潜在影响**: 边界条件（如嵌套 multipart、非 ASCII 文件名、CRLF 变体）可能处理不完整。

**建议修复**: 如果 multipart 使用场景扩展，考虑引入 `multer` 或 `multipart` crate。当前有限使用场景下风险可控。

---

### 【低危-4】`lifecycle/shutdown.rs` — 关机请求通过 HTTP Header 传递 RPC Token

**文件**: `crates/service/src/lifecycle/shutdown.rs`

**问题原因**: 优雅关机通过向本地 TCP 端口发送 HTTP 请求实现，RPC token 放在 `X-CodexManager-Rpc-Token` header 中。虽然目标是 localhost，但如果进程间有 HTTP 代理拦截，token 可能泄漏。

**潜在影响**: 极端场景下的 token 泄漏风险。

**建议修复**: 当前设计为 localhost 通信，风险可控。记录设计假设即可。
