# Code Review 5 — Copilot 自定义资产

## Findings

### [HIGH] 自定义 agent 的 tools 标识与当前 VS Code Copilot 工具命名不一致，核心能力声明存在失效风险

文件：

1. [.github/agents/context7.agent.md](.github/agents/context7.agent.md#L5)
2. [.github/agents/frontend-nextjs.agent.md](.github/agents/frontend-nextjs.agent.md#L5)
3. [.github/agents/tauri-bridge.agent.md](.github/agents/tauri-bridge.agent.md#L5)
4. [.github/agents/rust-gateway.agent.md](.github/agents/rust-gateway.agent.md#L5)
5. [.github/agents/playwright-ui.agent.md](.github/agents/playwright-ui.agent.md#L5)
6. [AGENTS.md](AGENTS.md#L70-L84)

Issue：`read/problems`、`agent/runSubagent` 以及 `context7Docs/*`、`uiTesting/*` 这类写法，与当前 Copilot 已公开和扩展内部可见的工具命名体系不一致。前者看起来像来自其他 agent harness 的路径式命名，后者则把 MCP server 名直接写成通配工具名；而 Copilot 当前对 MCP 工具的引用方式更接近精确的 `server/tool` 标识，而不是 `server/*`。这意味着这些 `.agent.md` 文件即使能被发现，也不代表 agent 真能拿到文档中声明的诊断、子代理委派和 MCP 调用能力。

Impact：仓库在 [AGENTS.md](AGENTS.md#L70-L84) 第 6.3 节声明的 `Context7-Expert`、`Frontend-Nextjs`、`Tauri-Bridge`、`Rust-Gateway`、`Playwright-UI` 工作流会与真实可执行能力脱节。最直接的结果是：触发 agent 后看起来“被选中了”，但具体工具可能不会出现或不会按预期授权，导致文档查询、问题诊断、子代理 handoff、Playwright MCP 验证这些核心路径失效。

Fix：把 `tools` 中的非标准标识改成当前 Copilot 支持的精确工具名或工具集名；MCP 能力改为逐个声明真实工具，而不是使用 `context7Docs/*`、`uiTesting/*` 这种 server 级通配；修改后需要在 VS Code 中实际触发一次每个 agent，验证工具面板里出现的能力与文档一致。

## No Additional Findings

- 目录位置本身符合当前 VS Code / GitHub Copilot 约定：根 [AGENTS.md](AGENTS.md)、[.github/agents](.github/agents)、[.github/instructions](.github/instructions)、[.github/skills](.github/skills)、[.vscode/mcp.json](.vscode/mcp.json) 都处在可发现位置。
- instruction 与 skill 的 frontmatter 结构未见明确阻断项。当前使用的 `name`、`description`、`applyTo`、`argument-hint`、`agents`、`handoffs` 组合没有发现会直接导致文件无法被解析的明显问题。
- 与仓库事实的对齐总体正常。第 6 节提到的多数路径、脚本、Cargo 包名与当前仓库一致，例如 [apps/package.json](apps/package.json)、[scripts/tests/gateway_regression_suite.ps1](scripts/tests/gateway_regression_suite.ps1)、[apps/src/lib/api/transport.ts](apps/src/lib/api/transport.ts)、[apps/src-tauri/src/commands/registry.rs](apps/src-tauri/src/commands/registry.rs)。
