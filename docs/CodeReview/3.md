## Findings

### 【严重-C1】`crates/core/src/crypto.rs:45-63` + `crates/service/src/storage/storage_helpers.rs:175-186` + `crates/service/src/lifecycle/startup.rs:18-19` + `crates/service/src/rpc_dispatch/mod.rs:78-93` — 加密初始化失败仍可能导致敏感数据 fail-open

- **问题原因**：`encrypt_field()` 在加密模块未初始化或加密失败时仍会返回可落库字符串；同时启动阶段和 `initialize` RPC 对存储初始化失败没有 fail-closed。
- **潜在影响**：一旦密钥文件缺失、损坏或初始化异常，后续账号令牌与平台密钥可能以明文形式写入 SQLite，而服务表面上仍继续提供写接口。
- **建议修复**：让加密初始化与加密操作显式返回 `Result`，并在启动、`initialize`、敏感写路径上统一 fail-closed；初始化失败时拒绝 secret 写入。

### 【高危-H1】`crates/core/src/crypto.rs:75-87` + `crates/core/src/crypto.rs:110-123` — keyfile 异常时会静默切换到新内存密钥

- **问题原因**：现有逻辑对密钥文件异常长度、目录创建失败、文件写入失败、ACL 设置失败等情况没有强制中止，而是继续生成只存在于内存中的新密钥。
- **潜在影响**：进程重启后旧密文无法解开，表现为账号/密钥读取异常、数据等效丢失，且故障延迟暴露。
- **建议修复**：把 `load_or_create_key()` 改成显式失败路径；已有 keyfile 长度异常或权限设置失败时直接阻止启动。

### 【高危-H2】`crates/service/src/account/account_export.rs:109-156` + `crates/service/src/rpc_dispatch/account.rs:90-92` — 导出接口仍返回明文令牌

- **问题原因**：`account/export` 和 `account/exportData` 仍会把 `access_token`、`id_token`、`refresh_token` 以明文 JSON 形式导出；当前只增加了风险提示，没有密码学保护。
- **潜在影响**：任何可调用该 RPC 的调用方都能批量拿走全部账号令牌，且 `exportData` 还会把明文令牌带入 RPC/前端内存链路。
- **建议修复**：至少下线 `exportData` 的明文返回；理想方案是改为带导出密码的加密包导出。

### 【高危-H3】`crates/service/src/app_settings/api/mod.rs:62-66` + `crates/service/src/app_settings/env_overrides/process.rs:20-56` + `crates/service/src/usage/refresh/settings.rs:72-143` — 运行中仍通过 process env 承载可变配置

- **问题原因**：虽然 `set_var/remove_var` 已补 `unsafe` 包裹，但运行态 RPC 仍会在服务活跃时修改进程环境变量，`unsafe` 的并发前提并不成立。
- **潜在影响**：后台轮询、HTTP worker、keepalive 等并发读取环境变量时，可能出现未定义行为或配置快照不一致。
- **建议修复**：把运行时配置迁移到进程内状态结构体/原子变量；环境变量只保留为启动输入。

### 【高危-H4】`crates/start/src/main.rs:334-363` — Windows 启动器在 web 已运行分支可能杀掉刚拉起的 service

- **问题原因**：Windows 下 start 进程会先拉起 service 并纳入 JobObject；若随后检测到 web 已运行就直接退出，主进程退出会导致 JobObject 关闭并终止刚拉起的 service。
- **潜在影响**：在“web 还活着但 service 已掉线”的恢复场景下，重新运行启动器会得到一个看似恢复、实则 service 立即再次退出的坏状态。
- **建议修复**：若本次已拉起 service，则不要在 `web 已运行` 分支直接退出；应继续监督 service，或避免把该 service 绑定到随主进程退出即销毁的 JobObject。

### 【高危-H5】`crates/start/src/main.rs:334-335` + `crates/web/src/service_gateway.rs:52-102` — start/web 仍仅凭 TCP 连通性判断 service 已就绪

- **问题原因**：当前只要 TCP 能连上就视为 service 存在，没有像桌面端那样做 `initialize` 握手并确认 `server_name`。
- **潜在影响**：若目标端口被其他本地进程占用，启动器会误判 service 已运行；web 还可能把受保护 RPC 请求和 token 转发给错误目标。
- **建议修复**：把探测升级为轻量级握手校验，至少验证 `server_name == codexmanager-service`，失败时停止启动并明确提示端口冲突。

### 【高危-H6】`apps/src-tauri/src/commands/updater/prepare.rs:212-251` + `apps/src-tauri/src/commands/updater/apply.rs:401-490` — 桌面更新缺少完整性/签名校验

- **问题原因**：更新器下载 GitHub Release 资产后直接解压或执行，没有摘要校验、签名校验，也没有在准备更新和实际应用之间做二次校验。
- **潜在影响**：一旦发布产物被篡改或本地更新目录被污染，桌面端会直接执行恶意二进制。
- **建议修复**：为发布产物增加校验清单或签名文件；下载后与应用前都执行校验，平台允许时叠加系统签名验证。

### 【高危-H7】`.github/workflows/release-all.yml:30-31,53` + `.github/actions/setup-release-env/action.yml:44-78` — 发布工作流中的第三方 Action 未固定到 SHA

- **问题原因**：发布流水线拥有 `contents: write`，但仍通过 tag 形式引用第三方 Action，而不是固定完整 commit SHA。
- **潜在影响**：一旦上游 Action 发生 tag 漂移或供应链被污染，发布流程会在持有发布权限时执行未审计代码。
- **建议修复**：把第三方 Action 固定到完整 SHA，并建立统一升级流程。

### 【中危-M1】`apps/src-tauri/src/commands/updater/apply.rs:119,189` — 便携版覆盖更新不会清理已删除旧文件

- **问题原因**：Windows 使用 `Copy-Item -Recurse -Force`、Unix 使用 `cp -Rf` 覆盖复制，都会保留新版本已删除的历史文件。
- **潜在影响**：旧 DLL、脚本或辅助二进制可能长期残留，造成状态不一致甚至继续加载已废弃组件。
- **建议修复**：改成受管清单镜像同步，或“完整展开到新目录 + 原子切换”。

### 【中危-M2】`scripts/tests/chat_tools_hit_probe.ps1` + `scripts/tests/codex_stream_probe.ps1` + `scripts/tests/gateway_regression_suite.ps1` — 网关回归套件缺少关键语义断言退出码（已修复）

- **原问题**：probe 只输出观测结果，没有把 `tool_hit`、`restored_name_ok`、SSE 终止完整性、重复 chunk 风险等条件转成非零退出码。
- **修复**：`chat_tools_hit_probe.ps1` 与 `codex_stream_probe.ps1` 现已在关键语义不满足时直接 `exit 1`，`gateway_regression_suite.ps1` 会继续聚合失败结果。

### 【中危-M3】`apps/src-tauri/tauri.conf.json:8-9` + `scripts/rebuild-linux.sh:101-102` + `scripts/rebuild-macos.sh:114-115` — Linux/macOS 本地重建脚本仍依赖 Windows 风格 `beforeBuildCommand`

- **问题原因**：`tauri.conf.json` 中的 `beforeBuildCommand` 使用的是 Windows CMD 语法；CI 会先移除该字段，但 `rebuild-linux.sh` / `rebuild-macos.sh` 本地重建入口没有做同样处理。
- **潜在影响**：本地 Linux/macOS 构建可能直接失败，难以及时复现跨平台打包问题。
- **建议修复**：改成跨平台命令，或让本地重建脚本先执行与 CI 相同的 `disable-tauri-before-build` 逻辑。
