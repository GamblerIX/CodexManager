# Code Review 1

## 结论
本轮全量审查发现 4 个问题，其中 2 个高危、2 个中危。`pnpm run build:desktop` 通过，但 `cargo test -p codexmanager-service --quiet` 仍有 1 个现存失败，说明设置快照与监听模式的语义已经出现回归。

## Findings

### 1. 高危：账号与密钥明文落库
文件: `crates/core/migrations/001_init.sql:17-23`, `crates/core/src/storage/tokens.rs:6-24`, `crates/core/migrations/019_api_key_secrets.sql:1-6`, `crates/core/src/storage/api_keys.rs:258-279`

问题原因: `id_token`、`access_token`、`refresh_token`、`api_key_access_token` 以及 API key secret 都直接存入 SQLite，当前没有系统密钥库或加密封装。

潜在影响: 数据库文件一旦泄露，攻击者可以直接接管已登录账号和上游平台密钥，风险覆盖账号池、网关转发和平台 Key。

建议修复: secret 迁移到系统密钥库，只在 SQLite 中保存句柄、元数据或哈希；同时补历史数据迁移和脱敏清理。

### 2. 高危：监听模式快照与实际生效值不一致
文件: `crates/service/src/app_settings/api/current.rs:62-66`, `crates/service/tests/app_settings.rs:990-1001`, `apps/src/app/settings/page.tsx:1097-1105`

问题原因: 后端 `app_settings_get` 返回的是持久化监听模式 `serviceListenMode`，而环境变量覆盖场景下真实生效值来自 `serviceListenModeEffective`。前端下拉框直接绑定 `snapshot.serviceListenMode`，因此会把有效值显示错。

潜在影响: 当 `CODEXMANAGER_SERVICE_ADDR=0.0.0.0:4999` 之类的环境变量覆盖存在时，UI 仍可能显示“仅本机”，监听地址预览和重启提示也会误导用户。相关测试 `app_settings_get_loads_env_backed_dedicated_settings_when_storage_missing` 已失败。

建议修复: 要么后端在 env override 生效时返回 effective mode，要么前端改为以 `serviceListenModeEffective` 展示，并把持久化模式标成“配置值”。

### 3. 中危：`Authorization` Bearer 解析过于严格
文件: `crates/service/src/gateway/request/incoming_headers.rs:23-31,97-106,167-183`, `crates/service/src/gateway/request/tests/incoming_headers_tests.rs:4-21`

问题原因: 认证路径用的是 `strict_bearer_token()`，只接受精确的 `Bearer ` 前缀；而同一模块的 sticky 逻辑才使用大小写不敏感解析。两套逻辑已经分叉。

潜在影响: 合法客户端如果发出 `authorization: bearer ...` 或中间层改写了大小写，网关会误判为缺少 API key，造成协议兼容性回归。

建议修复: 统一改成大小写不敏感的 Bearer 解析，避免认证与粘性路由继续漂移。

### 4. 中危：Web 访问密码哈希强度不足
文件: `crates/service/src/auth/web_access.rs:17-26,48-70`

问题原因: 当前使用的是 `sha256$salt$hash` 形式的单轮哈希，只有随机盐，没有引入 Argon2、PBKDF2 或 scrypt 这类带成本参数的 KDF。

潜在影响: 一旦 web 访问密码哈希被获取，离线撞库成本偏低；对可能暴露在局域网的管理入口来说，这个强度不够。

建议修复: 升级为 Argon2id 或至少 PBKDF2/scrypt，并使用带版本号的存储格式，便于后续平滑迁移。
